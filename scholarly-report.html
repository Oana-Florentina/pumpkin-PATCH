Abstract

PhoA (Phobia Web Assistant) is a multi-device web application designed to help users manage and overcome phobias through personalized remedies, context-aware alerts, and semantic web integration. The system leverages DBpedia and Wikidata knowledge bases, implements RDF data modeling with Schema.org vocabulary, and provides real-time monitoring through device integration. Context-aware notifications are triggered based on environmental factors including time, location, weather conditions, and physiological sensors. The application follows modern web engineering practices with a microservices architecture deployed on AWS cloud infrastructure.

1. Introduction

Phobias affect millions of people worldwide, significantly impacting their quality of life. Traditional treatment approaches often lack personalization and real-time contextual awareness. PhoA addresses these limitations by combining semantic web technologies with modern cloud-based architecture to deliver intelligent, context-aware phobia management.

1.1 Motivation

The motivation for developing PhoA stems from the need for:

  • Personalized phobia management based on individual user profiles
  • Context-aware alerts triggered by environmental and physiological factors
  • Integration with semantic web knowledge bases for evidence-based remedies
  • Multi-user support for families, coworkers, and social networks
  • Real-time monitoring through wearable devices and sensors

1.2 Objectives

The primary objectives of this project are:

  1. Develop a semantic web-based application using RDF, SPARQL, and Schema.org
  2. Implement context-aware alert system based on time, location, and environmental data
  3. Integrate with DBpedia and Wikidata for phobia information and remedies
  4. Provide multi-device support with smartwatch integration
  5. Deploy a scalable microservices architecture on AWS cloud
  6. Implement rule-based reasoning for intelligent alert generation

2. Related Work

Several approaches exist for phobia treatment and mental health management:

  • Virtual Reality Exposure Therapy (VRET): Uses immersive VR environments for gradual exposure to phobic stimuli
  • Mobile Mental Health Apps: Applications like Headspace and Calm provide meditation and anxiety management
  • Wearable Health Monitoring: Devices tracking physiological responses to stress and anxiety
  • Semantic Health Systems: Knowledge-based systems using medical ontologies for diagnosis and treatment

PhoA differentiates itself by combining semantic web technologies with real-time context awareness and multi-user support, providing a comprehensive solution for phobia management.

3. System Architecture

3.1 Overview

PhoA follows a microservices architecture with clear separation between frontend, backend services, and data storage. The system is deployed on AWS cloud infrastructure for scalability and reliability.

3.2 Components

3.2.1 Frontend

Built with React 18.2.0, the frontend provides:

  • Responsive web interface with mobile support
  • RDFa markup for semantic data representation
  • Real-time device data visualization
  • Dark mode with automatic theme switching based on phobias

3.2.2 Backend Services

Node.js/Express microservices deployed on AWS Lambda:

  • authService.js - AWS Cognito authentication
  • rdfService.js - RDF data conversion and Schema.org mapping
  • timeService.js - Time-based alert generation
  • weatherService.js - Open-Meteo API integration
  • sunriseSunsetService.js - Daylight tracking for nyctophobia
  • nominatimService.js - Location type detection

3.2.3 Data Storage

AWS DynamoDB for NoSQL data storage with tables for:

  • Users and authentication
  • Phobia definitions with Wikidata IDs
  • User-phobia associations
  • Alert history and context data

4. Semantic Web Integration

4.1 RDF Data Modeling

All phobia data is modeled using Schema.org vocabulary with the following types:

  • MedicalCondition - Phobia definitions
  • MedicalObservation - Health monitoring data
  • MedicalTherapy - Treatment recommendations
  • Drug - Medication information

Example RDFa Markup:

<div vocab="http://schema.org/" typeof="MedicalCondition">
  <h2 property="name">Arachnophobia</h2>
  <p property="description">Fear of spiders</p>
  <span property="code" typeof="MedicalCode">
    <meta property="codeValue" content="F40.210"/>
  </span>
</div>

4.2 SPARQL Queries

Five SPARQL queries are implemented for knowledge base integration:

Query 1: Phobia Information from DBpedia

PREFIX dbo: <http://dbpedia.org/ontology/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>
SELECT ?phobia ?label ?abstract
WHERE {
  ?phobia a dbo:Disease ;
          rdfs:label ?label ;
          dbo:abstract ?abstract .
  FILTER(CONTAINS(LCASE(?label), "phobia"))
  FILTER(LANG(?label) = "en")
}
LIMIT 10

Query 2: Medications from Wikidata

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
SELECT ?drug ?drugLabel ?indication
WHERE {
  ?drug wdt:P31 wd:Q12140 ;
        wdt:P2175 ?condition ;
        rdfs:label ?drugLabel .
  FILTER(CONTAINS(LCASE(?indication), "anxiety"))
}
LIMIT 20

Additional queries retrieve therapeutic procedures, allergy information, and symptom data. Full query documentation available in SPARQL_QUERIES.md.

5. Context-Aware Alert System

5.1 Rule-Based Reasoning

The system implements rule-based logic for intelligent alert generation:

Rule Example 1: Seasonal Allergies

IF phobia = "pollenAllergy" 
   AND season = "Spring" 
   AND pollenLevel = "High"
THEN 
   sendAlert("High pollen levels detected")
   suggestRemedy("Take antihistamine")

Rule Example 2: Nyctophobia (Fear of Darkness)

IF phobia = "nyctophobia" 
   AND timeOfDay = "Night"
THEN 
   forceTheme("light-mode")
   sendAlert("Nighttime detected. Keep lights on")

Rule Example 3: Claustrophobia

IF phobia = "claustrophobia" 
   AND roomSize = "Small"
THEN 
   sendAlert("Small space detected")
   suggestRemedy("Practice breathing exercises")

5.2 Context Data Sources

5.2.1 Temporal Context

  • Time of day (morning, afternoon, evening, night)
  • Day of week (weekend social event warnings)
  • Season (Spring, Summer, Fall, Winter)
  • Sunrise/sunset times (Sunrise-Sunset.org API)

5.2.2 Environmental Context

  • Weather conditions (Open-Meteo API)
  • Temperature, humidity, precipitation
  • UV index, cloud cover, wind speed
  • Weather code (WMO standard 0-99)

5.2.3 Location Context

  • GPS coordinates (browser geolocation)
  • Address and place type (Nominatim API)
  • Nearby amenities (hospitals, bridges, tunnels)

5.2.4 Physiological Context

  • Heart rate monitoring (simulated smartwatch)
  • Altitude tracking
  • Noise level detection
  • Room size estimation

7. Implementation Details

7.1 External API Integration

All integrated APIs are free and require no authentication keys:

  • Open-Meteo: https://api.open-meteo.com/v1/forecast
  • Sunrise-Sunset.org: https://api.sunrise-sunset.org/json
  • Nominatim: https://nominatim.openstreetmap.org/reverse

7.2 Authentication

AWS Cognito provides secure user authentication with:

  • User registration and login
  • JWT token-based authorization
  • Password reset functionality
  • Session management

7.3 Deployment

The application is deployed using:

  • Frontend: AWS S3 + CloudFront
  • Backend: AWS Lambda with API Gateway
  • Database: AWS DynamoDB
  • CI/CD: GitHub Actions workflows

Abstract

PhoA (Phobia Web Assistant) is a multi-device web application that provides context-aware phobia detection and personalized remedies through semantic web technologies and rule-based inference. The system stores all application data in Apache Jena Fuseki RDF triple store using Schema.org vocabulary, queries Wikidata and DBpedia knowledge bases via SPARQL, and implements logical reasoning with Jena's GenericRuleReasoner for automated phobia detection. Context-aware alerts are triggered based on environmental sensors (location, weather, temperature, altitude), physiological data (heart rate, noise level), and social triggers from user groups. The application integrates multiple external APIs (Open-Meteo, Nominatim, Sunrise-Sunset.org, TV Tropes) and uses Google Gemini AI for automated rule generation. Deployed on AWS cloud infrastructure with Lambda functions, EC2 (Fuseki), S3, and CloudFront, the system demonstrates modern web engineering practices including microservices architecture, serverless computing, and semantic web technologies with RDF-native data management. The rule-based approach using Apache Jena enables dynamic, data-driven phobia detection with SPARQL-based rule loading and RDF inference, fulfilling the bonus requirement for logical rule-based systems.

1. Introduction

Phobias are anxiety disorders affecting millions of people worldwide, significantly impacting quality of life and daily activities. Traditional treatment approaches often lack personalization and real-time contextual awareness, relying on scheduled therapy sessions without considering immediate environmental triggers. PhoA addresses these limitations by combining semantic web technologies, rule-based inference systems, and cloud-based architecture to deliver intelligent, context-aware phobia management.

1.1 Motivation

The motivation for developing PhoA stems from several key challenges:

  • Lack of personalization: Generic treatment approaches don't account for individual triggers and contexts
  • No real-time detection: Traditional therapy can't respond to immediate phobia triggers
  • Limited knowledge integration: Existing solutions don't leverage semantic web knowledge bases
  • Isolated treatment: No support for family/friend groups to help identify triggers
  • Manual monitoring: Users must self-report symptoms without automated detection

PhoA solves these challenges through:

  • Rule-based inference with Apache Jena for automated phobia detection
  • Integration with Wikidata (156 phobias) and DBpedia (8 treatments) via SPARQL
  • Real-time sensor monitoring (heart rate, location, weather, noise)
  • Group-based trigger reporting and social support
  • AI-powered rule generation using Google Gemini

1.2 Objectives

The primary objectives of this project are:

  1. Develop a semantic web application using RDF, SPARQL, Schema.org, and RDFa markup
  2. Implement rule-based inference system with Apache Jena for phobia detection (bonus requirement)
  3. Integrate multiple knowledge bases (Wikidata, DBpedia, TV Tropes) via SPARQL queries
  4. Create context-aware alert system based on environmental and physiological sensors
  5. Provide multi-user support with group trigger reporting and QR code sharing
  6. Deploy scalable microservices architecture on AWS cloud (Lambda, EC2, S3, CloudFront)
  7. Generate sensor rules automatically using AI (Gemini 2.5 Flash)
  8. Aggregate remedies from multiple sources (DBpedia treatments, NHS guides, Reddit communities, TV Tropes media)
  • Implement rule-based reasoning for intelligent alert generation

    4.3 Rule-Based Inference with Apache Jena

    PhoA implements a logical rule-based system using Apache Jena GenericRuleReasoner for automated phobia detection. Rules are stored as schema:PropertyValueSpecification in Fuseki RDF triple store and dynamically loaded via SPARQL queries for inference.

    4.3.1 AI-Powered Rule Generation with Google Gemini

    Rules are automatically generated using Google Gemini 2.5 Flash with specialized prompt engineering: 

    ACT AS A SPECIALIST PSYCHOLOGIST AND DATA ENGINEER.
CONTEXT: Phobia detection system. Map "${phobiaName}" (Trigger: "${mainTrigger}") to sensors.
TASK: Imagine a panic episode and determine sensor thresholds.

AVAILABLE SENSORS:
- location_type: hospital, school, park, forest, apartments, bridge...
- weather_code: 0 (clear), 3 (cloudy), 45 (fog), 61 (rain), 71 (snow), 95 (storm)
- temperature: -30 to 50 °C
- altitude: 0 to 3000 m
- heart_rate: 40 to 200 BPM
- noise_level: 0 to 120 dB
- is_night: true or false
- season: Spring, Summer, Fall, Winter

OUTPUT: 7 Schema.org PropertyValue objects with sensor thresholds

    Gemini analyzes phobia psychology and generates contextually appropriate sensor rules. For example, claustrophobia receives high heart_rate (110 BPM), confined location_type (elevator), and low noise_level thresholds.

    4.3.2 RDF Rule Storage in Fuseki

    Generated rules are stored in Fuseki as RDF triples: 

    @prefix schema: <http://schema.org/> .
@prefix phoa: <http://phoa.com/> .

phoa:rule/Q220783 a schema:PropertyValueSpecification ;
    schema:name "arachnophobia" ;
    phoa:mainTrigger "spider" ;
    phoa:location_type "forest" ;
    phoa:heart_rate 110 ;
    phoa:season "Summer" ;
    phoa:noise_level null .

    4.3.3 Jena Reasoner Inference Process

    1. Load Facts: User phobias, sensor data, and group messages inserted into Jena Model as RDF triples
    2. Query Rules from Fuseki: SPARQL queries retrieve sensor conditions for each user phobia
    3. Build Jena Rules: Dynamic rule construction from Fuseki data: 
      [rule_Q220783: 
  (?u schema:medicalCondition phoa:phobia/Q220783)
  (?u phoa:heart_rate ?h) ge(?h, 100)
  (?u phoa:groupText ?txt) regex(?txt, '.*spider.*', 'i')
  -> (?u phoa:needsAlert 'Q220783')]
    4. Apply Reasoner: GenericRuleReasoner performs forward-chaining inference
    5. Extract Alerts: Query inferred phoa:needsAlert statements
    6. Retrieve Recommendations: SPARQL query to Fuseki for treatments linked to detected phobia

    4.3.4 Example: Claustrophobia Detection

    Input Facts: 

    phoa:user/current schema:medicalCondition phoa:phobia/Q186892 .
phoa:user/current phoa:heart_rate 100 .
phoa:user/current phoa:groupText "confined space" .

    Rule from Fuseki: 

    phoa:rule/Q186892 phoa:mainTrigger "confined space" ;
                  phoa:heart_rate 100 .

    Jena Inference: Reasoner matches conditions → infers phoa:user/current phoa:needsAlert 'Q186892'

    Output: Alert with recommendations from Fuseki treatments

    5. Context-Aware Alert System

    5.1 Rule-Based Reasoning with Apache Jena

    The system implements a logical rule-based approach using Apache Jena for intelligent alert generation. Rules are dynamically loaded from Fuseki RDF triple store via SPARQL queries and applied through Jena reasoner inference.

    5.1.1 Rule Structure (Schema.org PropertyValue)

    {
  "phobiaId": "Q886731",
  "phobiaName": "blood phobia",
  "mainTrigger": "blood",
  "sensorRules": [
    {"@type": "PropertyValue", "name": "location_type", "value": "hospital"},
    {"@type": "PropertyValue", "name": "heart_rate", "value": 120, "unitText": "BPM"},
    {"@type": "PropertyValue", "name": "temperature", "value": 18, "unitText": "°C"},
    {"@type": "PropertyValue", "name": "noise_level", "value": 55, "unitText": "dB"}
  ]
}

    5.1.2 Inference Logic

    Alert generation follows this logic:

    1. Sensor Matching: Compare current sensor values with rule thresholds using tolerances (±10 BPM, ±5°C, ±15 dB, ±20m)
    2. Text Matching: Check if mainTrigger appears in group messages
    3. Decision:
      • If mainTrigger exists: Alert = (≥60% sensors match) AND (text match)
      • If no mainTrigger: Alert = (≥60% sensors match)
    4. Recommendations: Retrieve possibleTreatment from phobia data

    5.1.3 Example Scenario

    User: Has blood phobia (Q886731)
Context: location_type="hospital", heart_rate=115, temperature=20
Group: Someone wrote "I saw blood"

Matching:
  ✓ location_type: "hospital" == "hospital"
  ✓ heart_rate: 115 in range [110, 130] (120 ±10)
  ✓ temperature: 20 in range [13, 23] (18 ±5)
  → 3/4 sensors = 75% > 60% threshold
  ✓ text: "blood" found in message

Result: ALERT with recommendations:
  - Practice deep breathing
  - Exposure therapy (from DBpedia)
  - NHS Health Guide

    8. Bonus Implementation: Rule-Based System with Apache Jena

    8.1 Overview

    PhoA implements a logical rule-based system using Apache Jena, fulfilling the bonus requirement for "providing various Description Logics reasoning features by using a knowledge-based reasoner tool and/or a logical rule-based system (e.g. Apache Jena)".

    8.2 Implementation

    8.2.1 RDF Model Construction

    The Jena reasoner constructs an RDF model with user facts: 

    Model model = ModelFactory.createDefaultModel();
Resource user = model.createResource(PHOA + "user/current");

// Add user phobias (Schema.org)
user.addProperty(model.createProperty(SCHEMA + "medicalCondition"), 
               model.createResource(PHOA + "phobia/Q220783"));

// Add sensor facts directly on user
user.addLiteral(model.createProperty(PHOA + "heart_rate"), 100L);
user.addProperty(model.createProperty(PHOA + "location_type"), "forest");
user.addProperty(model.createProperty(PHOA + "groupText"), "i saw a spider");

    8.2.2 Dynamic Rule Loading from Fuseki

    Rules are loaded from Fuseki RDF triple store at runtime via SPARQL queries, not hardcoded:

    • SPARQL query to Fuseki for rules matching user phobias
    • Extract sensor conditions (mainTrigger, heart_rate, location_type, etc.)
    • Build Jena rules dynamically with regex and numeric comparisons
    • Apply GenericRuleReasoner for forward-chaining inference

    8.2.3 Inference Logic

    The rule-based approach evaluates: 

    IF (user has phobia X) 
   AND (≥60% sensors match rule thresholds)
   AND (mainTrigger in group messages OR no mainTrigger)
THEN
   Generate alert with recommendations

    8.2.4 Benefits

    • Declarative: Rules defined as data, not code
    • Dynamic: Rules can be modified without code changes
    • Semantic: RDF triples enable reasoning and inference
    • Scalable: New phobias and rules added via database

    The system leverages transfer learning by using Gemini's pre-trained medical knowledge without requiring custom dataset collection or training infrastructure. The LLM has been trained on vast medical literature and can generate realistic sensor thresholds based on psychological and physiological patterns.

    4. Validation: Rules stored as Schema.org PropertyValueSpecification in Fuseki RDF store
